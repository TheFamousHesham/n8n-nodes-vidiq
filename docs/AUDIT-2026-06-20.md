# Security & Quality Audit — n8n-nodes-vidiq (2026-06-20)

Multi-agent audit of the node: parallel **security**, **n8n-compliance**, and **correctness**
reviews plus a **live end-to-end** run against the real vidIQ MCP server, with every finding handed
to an independent adversarial verifier (instructed to refute). 24 agents, ~20 findings, all
verified.

## Result: PASS

No security vulnerabilities. All confirmed correctness/compliance issues fixed. Strict verification
lint clean, build clean, 25 unit tests, live E2E green.

## Live E2E (real server, `https://mcp.vidiq.com/mcp`)

Drove the **built `dist` transport** through a fake n8n ctx that injects auth exactly like the
credential. Tested **25 non-generative tools** once each with minimal valid args, chaining real IDs
from `youtube_search`. The 14 generative/credit-heavy Studio + "watch" tools were **skipped to
protect credits** (their network path is the same shared transport, already exercised).

- **24/25 returned real structured data.** The single error (`channel_analytics`) is **benign
  authorization** — analytics are only available for channels the key owns; we passed a third-party
  channel. The server accepted our argument shape and reached business logic.
- **Zero argument-mapping failures** — every field name / enum / required / array shape the node
  produces was accepted by the live server.
- SSE parsing + `structuredContent` extraction worked for every tool (the two IG outlier tools
  return a plain-text content block, correctly surfaced as `{ text }`).
- **Credits: 4939 → 4839 (100 used; 20 paid calls × 5 credits).**

## Security — all clear

| Area | Verdict |
|---|---|
| Prototype pollution (user JSON → `buildArgs`/`Object.assign`) | **Not exploitable** — fresh object literals; `JSON.stringify` ignores prototype; no `for..in` merge into shared objects |
| JSON-RPC envelope injection | **Not possible** — `method`/tool `name` are hard-coded per operation; user input only nests under `params.arguments` |
| SSRF / unexpected outbound | **None** — the node only POSTs to the fixed vidIQ endpoint; URL fields are passed to vidIQ as data, never fetched by the node |
| Credential/secret leakage | **Node never attaches the key to errors**; redaction relies on the n8n framework (standard) |
| ReDoS | **None** — `isCreditError` regex and `toStringArray` are linear-time |
| Supply chain | **Zero runtime dependencies** (only `n8n-workflow` peer) |
| `eval`/`child_process`/`fs`/global timers | **None used** |

## Findings fixed

| Sev | Finding | Fix |
|---|---|---|
| **High** | `generate_video.duration` (required numeric) dropped at default `0` by the numeric-0-as-unset rule | default `8` + `required` + `minValue: 1` |
| **Medium** | Channel-search & IG outlier-reels `additionalFilters` (incl. signed growth %) folded into the 0-dropped bucket, so an explicit `0`/negative bound was lost | collection values merged as **literals** (a collection key exists only when the user adds it), preserving explicit `0`/negatives |
| **Low** | No property marked `required` even where the API requires it | added `required: true` to **43** catalog-required fields (binary-trio fields excluded) |
| **Low** | Credit-spending Studio ops exposed via `usableAsTool` without cost warning | appended "Consumes vidIQ credits." to 12 generative op descriptions (propagates into the AI-tool schema) |
| **Info** | SSE parser concatenated all `data:` lines (fragile if a keepalive/progress frame precedes the result) | parse **per-event**, return the JSON-RPC response frame; tolerates keepalives |

## Accepted with rationale (not changed)

- **Credential test relies on HTTP 401 for a bad key.** Verified empirically (`curl` with a bad key
  → `HTTP 401`). A `responseSuccessBody` rule was considered but rejected: vidIQ returns SSE
  (`text/event-stream`), which n8n's JSON body matcher cannot parse, so such a rule would be inert.
  The 401 path is the real and correct failure mode.
- **Numeric-0-as-unset for top-level filter fields** is intentional and correct: across the vidIQ
  surface, minimum filters at 0 are no-ops and offsets default to 0 server-side; only required
  numerics and user-explicit collection values needed special handling (both fixed above).
